Configuration Files

render.yaml

The "Blueprint" of our app. Render reads this file automatically to set up the environment, define the Python version, and set commands.

render.yaml
services:
  - type: web
    name: studyverse
    env: python
    # 1. How to install dependencies
    buildCommand: pip install --no-cache-dir -r requirements.txt
    
    # 2. How to start the live server
    startCommand: gunicorn --worker-class eventlet -w 1 --timeout 120 --bind 0.0.0.0:$PORT app:app
    
    # 3. Server OS Variables
    envVars:
      - key: PYTHON_VERSION
        value: 3.11.4

Procfile

Fallback/Legacy startup instruction tool used by Render (and Heroku) to know how to boot the application up after it finishes building.

Procfile
web: gunicorn --worker-class eventlet -w 1 --timeout 120 --bind 0.0.0.0:$PORT app:app
  • web: Defines this as a web process that accepts HTTP endpoints.
  • app:app Tells Gunicorn to look in app.py for the variable named app (our Flask instance).

The Step-by-Step Execution Journey

1. GitHub Push

Git Webhook Triggered

When you push code via git push origin main, GitHub sends a secure webhook (a silent HTTP POST request) to Render's servers alerting them that new code is present.

2. Build Phase

Container Virtual Environment

Render allocates a secure Linux container. It reads render.yaml, installs Python 3.11.4, and executes the buildCommand:

pip install -r requirements.txt

This pulls in Flask, Flask-SocketIO, SQLAlchemy, Eventlet exactly as defined, locking in versions.

3. Launch Phase

Gunicorn & Eventlet Initialization

Render fires the startCommand. Instead of Python's built-in single-threaded development server, we use Gunicorn engineered for production.

  • --worker-class eventlet: Crucial! Eventlet allows asynchronous I/O and GreenThreads. This is 100% required for our Byte Battles and WebSockets to work in real-time.
  • --timeout 120: Gives our app 120 seconds to process heavy tasks before forcefully resetting connection.
4. Flask Boot (app.py)

Internal App Initialization

As Gunicorn loads app.py, the Python interpreter starts at Line 1. This triggers a massive cascade of setup events:

  • SQLAlchemy binds: Application links to Render PostgreSQL via the SQLALCHEMY_DATABASE_URI env variable.
  • Blueprint Registration: Auth limits, Admin panels, and Routes are injected into the Flask app object.
  • SocketIO Wraps App: SocketIO takes control of server networking to allow realtime socket emission.
5. Traffic Routing

Live HTTP & WS Connections

The Application is now officially LIVE. At this stage:

1. User requests /dashboard.

2. Render Load Balancer accepts request on port 443 (HTTPS) and forwards it to our internal container $PORT.

3. Gunicorn receives it, hands it to an Eventlet worker.

4. Flask maps /dashboard to @app.route('/dashboard'), executes logic, querying DB, and uses Jinja2 to render HTML.

Environment Variables & Security

Our `app.py` looks for configuration secrets in the operating system environment. While locally we use .env files, on Render, these are securely stored in the Render Dashboard Settings.

DATABASE_URL

Overrides our local SQLite and connects our deployed app to the massive Render PostgreSQL Cluster instance dynamically.

SECRET_KEY

A 64-character encrypted string injected directly into the Flask config object to encrypt Cookies and Session data. Render hides this completely.

Deployment Simplified

Every time you push to GitHub, Render essentially creates a brand new, empty Linux computer, installs your dependencies, and runs your boot command. Nothing is persisted on the local file system; everything dynamic (like User Data, XP, Accounts) lives permanently in the external PostgreSQL database, separate from the code execution!
