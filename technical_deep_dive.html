1. Data Structures Implemented

We intentionally implemented custom data structures to optimize specific features.

Stack (LIFO)

File: app.py Lines: 962-981

Concept: Last-In, First-Out structure.

Usage: Implements the Undo Delete feature for Tasks. When you delete a task, it's pushed onto the stack. Clicking 'Undo' pops it off.

class Stack:
    def push(self, item):
        self._items.append(item) # O(1)

    def pop(self):
        return self._items.pop() # O(1)

LRU Cache

File: app.py Lines: 983-1018

Concept: Least Recently Used cache.

Usage: Used in ChatService to store recent AI responses. Prevents redundant API calls for repeated questions.

class LRUCache:
    def get(self, key):
        # Move accessed item to end (Most Recent)
        self.order.remove(key)
        self.order.append(key)
        return self.cache[key]

Hash Map (Dictionary)

File: app.py Variable: battles = {}

Concept: Key-Value store with O(1) access.

Usage: Stores active Battle Rooms state in memory. Uses `room_code` as key to instantly retrieve game state, players, and scores.

# In-memory storage for active battles
battles = {
    'AX92': {'state': 'active', 'players': [...]},
    'B7K1': {'state': 'waiting', 'players': [...]}
}

2. DBMS Concepts & PostgreSQL

ACID Properties

Atomicity, Consistency, Isolation, Durability

Where: Every `db.session.commit()` call in `app.py`.

Example: When buying an item, we deduct coins AND add the item. If one fails, SQLAlchemy rolls back BOTH to ensure data integrity.

Indexing

Where: `User` model (`app.py`)

email = db.Column(..., index=True)

Why: Creates a B-Tree index on the email column. Makes login lookups O(log n) instead of O(n) full table scan. Critical for performance.

Normalization (3NF)

Concept: Reducing redundancy.

Example: Instead of storing "Badge Name" in the `User` table explicitly, we have a `Badge` table and a `UserBadge` junction table. This avoids duplicate data if a badge name updates.

Joins & Relations

Where: `Todo` linked to `SyllabusDocument`.

todo.syllabus_id = ForeignKey('syllabus.id')

Why: Enables relational queries to finding all tasks belonging to a specific course document.

3. Security & SQL Injection

SQL Injection (SQLi) occurs when malicious SQL code is inserted into input fields to manipulate the database.

❌ Vulnerable (Bad)

Using f-strings allows attackers to execute commands.

# Attack: email = "' OR '1'='1"
query = f"SELECT * FROM user WHERE email = '{email}'"
cursor.execute(query)
# Result: Logs in without password!

✅ Secure (Our Code)

We use SQLAlchemy ORM which automatically parameterizes queries.

# SQLAlchemy handles escaping automatically
user = User.query.filter_by(email=email).first()

# Even with raw SQL, use params:
db.session.execute(
    text("SELECT * FROM user WHERE email=:e"), 
    {'e': email}
)

4. Deployment Architecture

Infrastructure

  • Platform: Render.com (Cloud PaaS)
  • Database: Managed PostgreSQL (Render)
  • Web Server: Gunicorn (Production WSGI)
  • Worker Class: Eventlet (Async for SocketIO)

CI/CD Pipeline

  1. Dev pushes code to GitHub.
  2. Render detects push via Webhooks.
  3. Render pulls latest code.
  4. Builds environment (`pip install -r requirements.txt`).
  5. Runs database migrations (`init_db_schema()`).
  6. Starts server (`gunicorn app:app`).
